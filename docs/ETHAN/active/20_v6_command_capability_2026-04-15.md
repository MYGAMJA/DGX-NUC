# Hylion v6 현재 정책의 명령 가능 범위 정리 (2026-04-15)

## 1) 결론

현재 Stage-B에서 학습 중인 Hylion v6 정책은 "목표 위치로 이동" 정책이 아니라 "base velocity command 추종" 정책이다.

즉, 내가 직접 넣을 수 있는 명령의 본질은 아래 3가지다.

1. 전후 속도 `lin_vel_x`
2. 좌우 속도 `lin_vel_y`
3. 제자리 회전 속도 `ang_vel_z`

따라서 "몇 초 동안 앞으로 가라", "몇 초 동안 왼쪽으로 이동하라", "몇 초 동안 돌아라" 같은 명령은 가능하다.

반대로 "저 좌표까지 가라" 또는 "저 방향으로 알아서 우회해서 걸어가라"는 현재 정책만으로는 직접 불가능하다. 그런 기능은 바깥쪽에 waypoint/planner/controller를 하나 더 둬서 위치 오차를 속도 명령으로 바꿔줘야 한다.

## 2) 코드 기준으로 지금 정책이 받는 명령

기본 명령 설정은 `δ3/hylion/env_cfg.py`의 `CommandsCfg.base_velocity`에 들어 있다.

현재 범위:

1. `lin_vel_x = (-0.5, 0.5)` m/s
2. `lin_vel_y = (-0.25, 0.25)` m/s
3. `ang_vel_z = (-1.0, 1.0)` rad/s
4. `heading = (-pi, pi)`
5. `resampling_time_range = (10.0, 10.0)` sec

의미:

1. 학습 중에는 하나의 명령 샘플이 기본적으로 10초 동안 유지된다.
2. 정책은 관측값으로 현재 `base_velocity` command를 같이 받는다.
3. 즉, 정책은 "현재 주어진 속도 명령을 따라 걷는 법"을 배우는 구조다.

## 3) 사용자가 직접 줄 수 있는 명령의 형태

### 가능한 것

1. 5초 동안 앞으로 0.2 m/s
2. 3초 동안 왼쪽으로 0.1 m/s
3. 2초 동안 제자리에서 0.5 rad/s로 회전
4. 앞으로 가면서 동시에 왼쪽으로 약간 이동
5. 앞으로 가면서 동시에 천천히 회전

예시 스케줄:

1. `0~5초`: `x=0.20, y=0.00, yaw=0.00`
2. `5~8초`: `x=0.15, y=0.10, yaw=0.00`
3. `8~10초`: `x=0.00, y=0.00, yaw=0.50`

이런 형태는 현재 정책 구조와 맞다.

### 바로 안 되는 것

1. "(2.3, -1.1) 좌표까지 가라"
2. "장애물을 피해서 저기로 가라"
3. "복도 끝까지 갔다가 90도 돌아라"를 고수준 명령만으로 처리
4. 자연어 문장을 바로 넣어서 정책이 해석하게 하는 방식

이런 기능은 locomotion policy 위에 별도 command governor 또는 planner가 필요하다.

## 4) 지금 이 정책이 실제로 검증된 범위

중요한 점은 "명령 인터페이스가 존재한다"와 "그 명령을 충분히 잘 따른다"는 별개라는 것이다.

현재까지 검증된 것은 주로 아래다.

1. contact signal이 살아났는지
2. `feet_air_time`이 비영값인지
3. PPO가 NaN 없이 버티는지
4. 조합 M1~M6 중 어떤 설정이 더 안정적인지

즉, 지금은 "정책이 죽지 않고 보행 학습이 가능한 상태"까지 복구한 단계다.
아직 "임의의 x/y/yaw 명령을 정확히 추종한다"를 체계적으로 측정한 단계는 아니다.

## 5) 사용자 질문에 대한 직접 답

### Q1. 학습시킨 다리로 내가 직접 몇 초 동안 어디로 걷게 할 수 있나?

답: 속도 명령 기준으로는 가능하다.

1. 몇 초 동안 앞으로 걷기: 가능
2. 몇 초 동안 옆으로 걷기: 구조상 가능
3. 몇 초 동안 회전하기: 구조상 가능
4. 앞으로 가면서 동시에 회전하기: 가능

단, "어디로"가 절대 좌표 의미라면 현재 정책 단독으로는 불가능하다. 현재 정책은 위치가 아니라 속도를 입력으로 받는다.

### Q2. 그럼 실제로 내가 원하는 경로를 따라가게 할 수 있나?

답: 가능은 하지만, 지금 당장은 locomotion policy만으로 끝나지 않는다.

필요한 구성:

1. 상위 경로/방향 명령기
2. 위치 오차를 `lin_vel_x`, `lin_vel_y`, `ang_vel_z`로 바꾸는 outer-loop controller
3. 속도 명령을 받아 실제 보행을 만드는 현재 locomotion policy

즉, 현재 학습은 "다리 레벨 보행 엔진"에 해당하고, 원하는 위치까지 보내는 건 그 위 레이어 작업이다.

## 6) 실사용 전에 꼭 필요한 추가 검증

아래 검증을 해야 "내가 몇 초 동안 어디로 걸으라고 시키는 것"이 실제로 된다고 말할 수 있다.

1. command tracking eval
- 고정 명령 세트별 추종 오차 측정
- 예: `(0.2, 0, 0)`, `(0, 0.1, 0)`, `(0, 0, 0.4)`, `(0.15, 0.05, 0.2)`

2. hold-time eval
- 2초, 5초, 10초 동안 명령 유지 시 넘어지지 않는지 확인

3. transition eval
- 전진 -> 정지 -> 측면 이동 -> 회전 전환 시 안정성 확인

4. real-robot safety gate
- yaw rate 제한
- lateral speed 제한
- emergency stop
- command smoothing

## 7) 현시점의 현실적인 판단

1. 전진 보행 명령은 가장 먼저 붙일 수 있다.
2. 측면 이동과 회전은 정책 구조상 지원되지만, 아직 품질 검증이 부족하다.
3. "좌표까지 이동"은 별도 outer loop 없이는 안 된다.
4. 따라서 현재 단계에서 가장 먼저 목표로 삼아야 할 것은 아래 순서다.

첫 단계:
1. 전진/정지 command tracking 검증
2. 제자리 yaw 회전 검증
3. 작은 lateral command 검증

그 다음 단계:
1. waypoint -> velocity command 변환기 추가
2. 실제 로봇용 safety clamp 추가
3. sim-to-real 보정

## 8) 한 줄 요약

현재 학습 정책으로 "몇 초 동안 앞으로/옆으로/회전"은 속도 명령 형태로 줄 수 있다. 하지만 "저 위치까지 가라"는 아직 아니고, 그 위에 command scheduler 또는 planner를 하나 더 얹어야 한다.
