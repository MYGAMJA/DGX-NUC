## 1. Stage C1 무게 랜덤 변동 — 실제 동작 확인

**대상:** `base` 링크(몸통)만, 다리·팔 링크는 건드리지 않음

```python
# env_cfg.py
"asset_cfg": SceneEntityCfg("robot", body_names="base"),
"operation": "add",
```

**USD 무게와의 관계:**
```
시뮬레이션 base 실제 무게 = USD 정의값 + 추가분
```
다리, 팔 등 나머지 링크는 USD 값 그대로 사용됨.

**브리핑 문서 수정 사항:**

| 항목 | 브리핑 기재 | 실제 동작 |
|------|------------|----------|
| 변동 방식 | ±0.5kg 랜덤 변동 | +0.5kg **고정** 추가 |

```python
# env_cfg_BG.py — C1 실제 설정
# HYLION_BASE_MASS_ADD_KG=0.5 → (0.5, 0.5) 균등분포
# 범위가 같으므로 랜덤이 아닌 고정값
self.events.add_base_mass.params["mass_distribution_params"] = (0.5, 0.5)
```

## 2. 속도 범위 확장 가능 여부

**현재 범위 (`env_cfg.py`):**
```python
lin_vel_x=(-0.5, 0.5),    # 전후 이동 (m/s)
lin_vel_y=(-0.25, 0.25),  # 좌우 이동 (m/s)
ang_vel_z=(-1.0, 1.0),    # 제자리 회전 (rad/s)
heading=(-math.pi, math.pi),
```

**기술적으로는 가능하지만, 지금 시점에서는 권장하지 않음.**

### 이유 1 — 학습 난이도 중첩
- 속도 범위 확장 = 동작 공간 확장 → 학습 부담 증가
- 현재 D1~D5 외력 강건성 학습 중간
- 두 가지 어려운 문제를 동시에 푸는 것은 비효율적

### 이유 2 — 보상 파라미터 재조정 필요
```python
# 현재 보상 std=0.25는 ±0.5 m/s 범위에 맞춘 값
track_lin_vel_xy_exp = RewTerm(
    params={"std": 0.25},   # 범위 바뀌면 이것도 조정 필요
    weight=2.0,
)
```
범위를 2배로 늘리면 `std`도 같이 재조정해야 보상 정규화가 제대로 됨.

### 이유 3 — 토크 한계 충돌 가능
관절 토크 패널티(`dof_torques_l2`)와도 충돌 가능. 빠른 속도에서 더 큰 토크가 요구됨.

### 올바른 확장 순서
```
D5 완료 (외력 강건성 달성)
  ↓
E 단계: 속도 범위 확장
  lin_vel_x = (-1.0, +1.0) m/s
  lin_vel_y = (-0.5, +0.5)  m/s
  std 등 보상 파라미터 재조정 병행
```

## 3. 외력 강건성이 꼭 필요한 이유

### ① 실제 로봇 전환 시 필수 조건
시뮬레이터는 완벽히 조용한 환경이지만, 현실에서는 에어컨 바람·케이블 저항·바닥 진동 등 시시각각 작은 외력이 발생함. 학습하지 않으면 실제 로봇으로 올렸을 때 가장 먼저 넘어지는 원인이 됨.

### ② 데모 시연 신뢰성
"밀어도 안 넘어진다"는 장면이 보행 AI의 완성도를 가장 직관적으로 보여주는 장면. 연구자가 살짝 건드리는 것만으로도 넘어지면 시연의 신뢰도가 크게 떨어짐.

### ③ 두 발 보행의 본질적 불안정성
두 발 보행은 물리적으로 항상 불안정한 시스템 (도립진자와 동일한 원리). 외력이 없어도 관절 마찰, 지면 불균일 등 내부 교란이 계속 발생함.

```
외력 학습 = 모든 교란에 대한 복원력(robustness)을 전반적으로 높이는 과정
```
